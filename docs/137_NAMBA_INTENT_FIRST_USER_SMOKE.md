# Namba Intent First-User Smoke

## Current status

- v0.5.1 release: published and verified
- v0.6.0 release: pre-release; superseded by docs/140 post-release verification

## Smoke goal

This smoke verifies pre-release current-tree Namba Intent onboarding from a temporary
PATH-resolved `namba-intent` binary. It does not verify public v0.6.0 install
or any hosted release artifact. Public v0.6.0 install proof is recorded later
in [docs/140 post-release verification](140_V0_6_0_POST_RELEASE_VERIFICATION.md).

## Decision

NAMBA_INTENT_FIRST_USER_SMOKE_PASS_WITH_NOTES

Notes:
- Current-tree `namba-intent` command-name, first-user init, first-user status,
  repeated init, lockfile safety, and deprecated `ni` shim checks passed.

  actors/outcomes, delivery surface, and the first open blocker question were
  still unresolved.

`namba-intent init . --yes` created the planning skeleton. The second
`namba-intent init . --yes` reported:

- `existing Namba Intent planning files found; namba-intent init will not overwrite them.`
- `adding missing files only.`
- existing planning docs and `.ni` files as `exists` and `unchanged`
- `created files: none`

## Lockfile safety

In `/tmp/namba-intent-first-user-smoke/lock.0JtvKg`, init created a blank
planning workspace. A fixture `.ni/plan.lock.json` was then copied into that
temporary project only. Its SHA-256 before the second init was:

```text
61d7265d58137893995f2a693b83d213874ab5a2028ecb693857350d0eb7c16c
```

The second `namba-intent init . --yes` reported:

```text
warning: .ni/plan.lock.json already exists; this project is already locked.
Use namba-intent status --proof --next-questions, then the amend/relock flow for locked planning changes.
No files changed by namba-intent init; the lockfile was not modified.
```

The SHA-256 after the command was unchanged:

```text
61d7265d58137893995f2a693b83d213874ab5a2028ecb693857350d0eb7c16c
```

No `namba-intent end` was run on the project root, and no root relock occurred.
