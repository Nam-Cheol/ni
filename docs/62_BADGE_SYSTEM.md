# Badge System

Task 103 adds a small README badge system for language links and factual trust
signals. The badge area must stay factual, local where practical, and aligned
with files or public repository state that actually exists.

## Badge Area

The README top section has two badge rows:

| Row | Source | Purpose |
| --- | --- | --- |
| Language chips | `assets/badge-english.svg`, `assets/badge-korean.svg` | Link between English and Korean README files. |
| Trust badges | Local files plus shields-style image URLs | Show only repository facts that are currently true. |

The language chips are generated local SVG files. Their text is short and is
also present in `alt` text and accessible link labels, so the language choice
is not available only through rendered pixels.

## Factual Badge Rules

Only add a factual badge when its backing evidence exists:

| Badge | Evidence |
| --- | --- |
| License | `LICENSE` exists and names the license being shown. |
| CI | A workflow exists under `.github/workflows/`. |
| Security | `SECURITY.md` exists. |
| Release | A public GitHub Release exists. |
| Docs | A docs index exists, currently `docs/00_START_HERE.md`. |

Do not add release badges for tags, drafts, local release notes, or planned
release work. A release badge means a public GitHub Release exists.

## Asset Checks

The language chip SVGs are generated by `scripts/render-assets.py` and required
by `scripts/check-assets.py`. Run these checks after badge changes:

```bash
python3 scripts/render-assets.py
python3 scripts/check-assets.py
bash scripts/quality.sh
```

Manual README inspection should confirm:

- the language chips are visible and link to the correct README files;
- Korean text renders acceptably in `assets/badge-korean.svg`;
- factual badges match actual repository evidence;
- any important trust claim is repeated in nearby text or accessible labels, not
  only inside badge images.
